### 🎯 리뷰 목적
이 Pull Request의 변경된 코드를 아래 기준에 따라 분석하고, 개선이 필요한 부분을 구체적으로 제안해 주세요.  
리뷰는 **가독성·일관성·유지보수성·성능·보안**을 기본으로 하되,  
**테스트 및 문서화 항목은 요청이 있을 때만 검토합니다.**

---

### ✅ 코드리뷰 기준

#### 1. 가독성 (Readability)
- 변수명, 메서드명, 클래스명이 명확하고 역할이 드러나는가?
- 불필요하게 중복되거나 복잡한 구조가 없는가?
- 조건문, 로직이 쉽게 이해될 수 있는가?

#### 2. 일관성 (Consistency)
- 기존 프로젝트의 코딩 컨벤션과 일치하는가?
- 동일한 기능 구현 패턴이 일관되게 유지되는가?

#### 3. 유지보수성 (Maintainability)
- 메서드/클래스가 단일 책임 원칙(SRP)을 지키는가?
- 중복 코드, 하드코딩, 매직넘버가 없는가?
- 예외 처리, 로깅, 에러 메시지가 명확하게 되어 있는가?

#### 4. 성능 및 비동기 처리 (Performance / Async)
- 비동기 로직에서 적절한 타임아웃, 예외처리, 쓰레드 제어가 있는가?
- 데이터 구조와 알고리즘이 효율적인가?
- 불필요한 DB 호출이나 반복 연산이 없는가?

#### 5. 보안 (Security)
- 입력값 검증(Input Validation)이 충분한가?
- 민감정보(email, token 등)가 로그나 응답에 노출되지 않는가?
- 인증/인가 로직이 안전하게 처리되는가?

---

### 🧩 선택적 리뷰 항목 (Optional)

#### 🧪 테스트 (Testability) — *요청된 경우만 검토*
- 단위 및 통합 테스트가 존재하는가?
- 주요 예외(NoSuchUserException 등)에 대한 테스트가 포함되어 있는가?
- mock/stub 사용이 적절한가?

#### 📘 문서화 (Documentation) — *요청된 경우만 검토*
- 변경된 코드에 주석 또는 Javadoc이 반영되었는가?
- API 변경 시 Swagger/OpenAPI 문서가 갱신되었는가?
- README 또는 ADR 등 문서가 필요한 경우 업데이트되었는가?

### 🧩 리뷰어 행동 가이드
Copilot은 아래 형식으로 리뷰를 작성해주세요:

**[요약]**
> 변경 내용의 핵심을 한 문장으로 설명

**[개선할 점]**
> 구체적인 개선 제안 (라인/메서드 단위로)

**[우선순위별 피드백]**
> 🟥 Critical – 기능 오류, 보안 문제  
> 🟧 Major – 성능/비동기/구조 개선 필요  
> 🟨 Minor – 스타일, 네이밍, 주석 개선  
> 🟩 Info – 참고용 제안

**[최종 코멘트]**
> 전반적인 코드 품질에 대한 종합 평가 및 다음 단계 제안  
> 예시:
> - “전체적으로 안정적인 구조이며, 일부 예외 처리 로직 보완이 필요합니다.”
> - “테스트 커버리지는 충분하나, 캐시 무효화 케이스 추가 검토가 좋겠습니다.”
> - “비동기 호출 흐름이 명확하나, 타임아웃 제어 부분을 좀 더 세밀하게 조정할 필요가 있습니다.”
